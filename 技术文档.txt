一、项目概述

1.1 项目背景

本系统是基于语音识别与自然语言处理技术的会议智能分析平台，核心目标是实现会议内容的自动转写、智能分析和合规校验，提升会议管理效率和质量把控能力。

1.2 核心能力

- 实时语音转写：浏览器端麦克风采集，WebSocket流式传输，边录边转写
- 音频上传分析：支持wav/mp3/ogg/flac/m4a等格式上传，批量分析
- 说话人分离：多人会议自动区分不同发言者，支持自动估计说话人数量
- 关键词提取：自动抓取会议核心关键词，支持TF-IDF+词性加权
- 主题分析：基于语义相似度自动识别16类会议主题
- 会议摘要：三级降级摘要生成（Qwen2.5大模型 → mT5 → TextRank）
- 情绪分析：正面/中性/负面三分类，配合关键词细粒度分析
- 合规检查：知识库语义匹配 + 要点覆盖核查 + 风险内容检测 + 智能评分
- 知识库管理：支持政策/要点/风险关键词等多类型条目，状态启停控制
- 实时合规告警：转写过程中实时检测风险内容并推送告警

1.3 技术亮点

- 全开源技术栈，无付费API依赖，支持离线部署
- 中文优化：bge-small-zh-v1.5语义模型、jieba分词、Qwen2.5中文摘要
- 多级降级机制：所有AI能力均有降级方案，确保可用性
- 否定语境识别：避免将"禁止承诺保本保收益"等合规培训内容误判为风险
- 智能知识库匹配：自动选择最相关的知识库条目计算分数，避免不相关内容拉低评分

2.3 核心流程

2.3.1 上传音频分析流程

用户上传音频 → 保存文件 → 创建会议记录 → 异步线程处理
  → 音频预处理（降噪/带通滤波/语音增强/归一化）
  → Whisper语音识别（medium模型）
  → 说话人分离（谱聚类 + 轮廓系数/BIC估计）
  → 文本分析（关键词/主题/摘要/情绪）
  → 合规检查（语义匹配/要点覆盖/风险检测）
  → 保存结果 → 推送进度 → 完成

2.3.2 实时转写流程

用户点击开始 → WebSocket连接 → 加载Whisper small模型
  → 浏览器采集音频（16kHz）→ Base64编码发送
  → 服务端5秒缓存块 → Whisper转写 → 实时合规检查
  → 结果实时推送前端
  → 停止转写 → 保存音频文件 → 快速分析 → 保存会议记录

2.3.3 合规检查流程

转写文本 → 筛选active状态知识库条目
  → 计算与各政策条目的语义相似度 → 取最高相关条目
  → 智能匹配对应的必传要点模板
  → 要点覆盖率计算（含否定语境过滤）
  → 风险关键词检测（含否定语境过滤 + 语义风险检测）
  → 关键词命中统计
  → 加权求和 → 生成评分等级与建议

四、核心功能模块详解

4.1 音频预处理模块（audio_preprocessor.py）

4.1.1 功能说明

对输入音频进行预处理，提升语音识别准确率。支持完整处理链，每个环节都有降级方案。

4.1.2 处理流程

1. 格式统一：librosa加载，自动重采样到16kHz单声道
2. 降噪：noisereduce谱减法（首0.5秒估计噪声），失败降级到自研谱减法
3. 回声消除：80-7000Hz带通滤波（4阶Butterworth）
4. 语音增强：MMSE短时谱振幅估计（判决引导法估计先验信噪比）
5. 归一化：RMS归一化到-20dB，防削波

4.1.3 辅助功能

- format_time：秒转MM:SS.mmm格式
- detect_speech_segments：基于STFT能量的语音活动检测（VAD）
- get_audio_quality_report：对比处理前后的SNR和降噪效果

4.2 语音识别模块（whisper_utils.py）

4.2.1 功能说明

基于OpenAI Whisper的语音转写，支持繁简修复。

4.2.2 核心函数

- transcribe_with_fix：封装Whisper转写，自动修复繁体转简体问题
- fix_traditional_chinese：繁简转换与常见转写错误修复

4.2.3 模型策略

- 上传分析：使用medium模型（准确率优先）
- 实时转写：使用small模型（速度优先）
- 模型预热：服务启动时预加载，避免首次请求延迟

4.3 说话人分离模块（speaker_diarization.py）

4.3.1 功能说明

识别并分离不同说话人的发言内容。采用"优先pyannote，失败降级增强聚类"的策略。

4.3.2 技术方案

方案A：pyannote.audio（推荐，需HF_TOKEN）
- 使用 pyannote/speaker-diarization-3.1 管线
- 支持CUDA加速，自动检测设备

方案B：增强聚类算法（默认，离线可用）
- 特征提取：MFCC(20维) + delta + delta2 + chroma + 频谱质心 + 带宽 + 过零率 + RMS
- 特征加权：MFCC权重最高(1.0)，加重音色相关特征
- 语音活动检测：RMS阈值过滤静音帧，只对语音帧聚类
- 说话人数量估计：轮廓系数 + GMM BIC 联合估计
  - 轮廓系数：衡量聚类紧密度，越高越好
  - BIC：贝叶斯信息准则，越小越好
  - 联合判断：轮廓系数差距<0.02时，选BIC更优的
- 聚类算法：谱聚类（Spectral Clustering）
  - affinity='nearest_neighbors'，对非球形簇效果更好
  - 失败降级：GMM + 层次聚类，再降级：KMeans
- 后处理：
  - 非语音帧映射到最近说话人
  - 中值滤波（窗口5）平滑标签，减少抖动
  - 合并短于0.3秒的片段
  - 合并相邻同说话人片段

4.3.3 性能特点

- 自动估计说话人数量（2-5人）
- 支持指定说话人数量
- 60秒超时保护，避免过长音频卡死
- 3人TTS合成音频可正确分离（谱聚类方案）

4.4 文本分析模块（text_analyzer.py）

4.4.1 关键词提取（extract_keywords）

算法：TF-IDF + 词性加权 + 停用词过滤

流程：
1. jieba词性标注，只保留名词/动词/专有名词等有意义词性
2. 三级停用词过滤：通用停用词 + 领域停用词 + 长度过滤
3. TF计算：词频 / 总词数
4. IDF计算：基于句子分布（词出现的句子越少越重要）
5. 词性加权：专有名词1.5x / 名词1.3x / 动词1.1x
6. 按综合得分排序取Top N

4.4.2 主题分析（analyze_topic）

支持16类会议主题：工作汇报、项目讨论、问题解决、决策制定、进度跟进、计划安排、意见交流、培训学习、风险讨论、合规审查、财务预算、人事安排、客户服务、产品设计、技术方案、市场营销。

算法：语义相似度（主）+ 关键词匹配（辅）
- 语义方案：bge-small-zh-v1.5编码文本与主题名，余弦相似度
- 关键词方案：每个主题有10个关联关键词，统计命中数
- 融合：语义分70% + 关键词分30%
- 超时保护：10秒超时，超时降级到纯关键词匹配

4.4.3 摘要生成（generate_summary）

三级降级机制：

第一级：Qwen2.5-1.5B-Instruct（推荐，质量最高）
- 指令式对话模板，专业会议摘要系统提示词
- 要求第三人称、结构化、客观准确
- 幻觉检测：新闻式模式匹配 + 实体覆盖率校验
- 生成参数：temperature=0.3，num_beams=1，max_new_tokens=300
- CPU运行约30-40秒/次

第二级：mT5_multilingual_XLSum（备选）
- 多语言摘要模型，Seq2Seq架构
- 同样有幻觉检测保护
- 生成速度快于Qwen

第三级：TextRank（保底，纯Python无模型依赖）
- 智能分句：支持无标点文本的基于触发词分句
- TextRank算法：句子相似度图 + PageRank迭代
- 位置加权：开头句子1.3x，结尾1.1x
- 关键词加权：命中关键词的句子额外加分
- 第三人称转换：50+条人称转换规则
- 去重：句子签名 + 相似度去重
- 行动项提取：识别"需要/必须/决定/负责/计划"等引导的句子
- 质量过滤：过滤问候语、问句、过短句

摘要清洗：
- 开头/结尾口语词清理（50+条模式）
- 重复字/词/标点清理
- 繁简转换

4.4.4 情绪分析（analyze_sentiment）

基于情感词典 + 规则的细粒度情绪分析：
- 正面/中性/负面三分类
- 情感词打分 + 程度副词加权 + 否定词反转
- 细分类别：积极、消极、反对、抱怨等
- 纯Python实现，无额外依赖

4.5 合规检查模块（compliance_checker.py）

4.5.1 功能说明

核心模块，负责将转写内容与知识库进行语义比对，生成合规评分和报告。

4.5.2 评分权重（可配置）

| 维度 | 默认权重 | 说明 |
|------|----------|------|
| 语义相似度 | 40% | 转写内容与政策内容的语义相关度 |
| 要点覆盖率 | 30% | 必传要点的覆盖比例 |
| 风险检测 | 20% | 风险内容检测，发现即扣分 |
| 关键词匹配 | 10% | 政策关键词命中比例 |

4.5.3 评分等级

- ≥90分：优秀
- 75-89分：良好
- 60-74分：合格
- <60分：不合格

4.5.4 核心算法

语义相似度计算（compute_semantic_similarity）：
- 主方案：bge-small-zh-v1.5 + 余弦相似度
- 降级方案：jieba分词 + Jaccard相似度
- 空文本保护：空输入直接返回0.0
- 异常自动降级：任何异常都自动切换到简单相似度

智能知识库匹配：
- 按item_type分类：政策类(policy/meeting_spirit)、要点类(key_points/required)、风险类(risk_keywords/forbidden)
- 语义相似度排序：找出与转写内容最相关的政策条目
- 只取最高分计算语义相似度得分，避免不相关内容拉低分数
- 要点匹配：关键词匹配最多的要点模板 + 政策标题相似度匹配
- 关键词统计：只统计最相关政策的关键词

否定语境识别（is_negated_context）：
- 20+条否定词模式：禁止、不得、不能、不要、严禁、不允许等
- 窗口范围：关键词前15-30字符内检测否定词
- 正则匹配：双重验证（前缀扫描 + 正则模式）
- 应用场景：风险关键词检测 + 语义风险检测

语义风险检测（detect_semantic_risks）：
- 8类表述不当模式：保本保收益、零风险、忽悠客户等
- 5类消极负面模式：问题太多、没办法、反对意见等
- 每类配严重程度（high/medium/low）
- 同样应用否定语境过滤

实时合规检查（realtime_compliance_check）：
- 单段文本快速检查
- 返回风险告警 + 要点覆盖 + 时间节点
- 用于WebSocket实时转写过程中的即时告警

4.5.5 知识库条目类型

| 类型 | 用途 |
|------|------|
| policy / meeting_spirit | 政策文件/会议精神，用于语义相似度匹配 |
| key_points / required | 必传要点，用于覆盖率计算 |
| risk_keywords / forbidden | 风险关键词，用于风险检测 |

4.5.6 时间节点标记

- point_time_markers：必传要点在音频中出现的时间点
- risk_time_markers：风险内容在音频中出现的时间点
- 支持跳转播放定位（前端可扩展）

4.6 实时转写模块（realtime_transcriber.py）

4.6.1 架构设计

基于Flask-SocketIO的WebSocket实时通信，服务端RealtimeTranscriber类管理每个会话。

4.6.2 核心流程

1. 连接建立：客户端连接 → 服务端确认
2. 开始转写：客户端发送start_transcription事件
   - 加载知识库条目和评分权重
   - 创建RealtimeTranscriber实例
   - 生成音频文件保存路径
3. 音频传输：客户端持续发送audio_chunk（Base64编码的PCM数据）
   - 服务端解码为float32数组
   - 累积到5秒缓冲 → Whisper small模型转写
   - 实时合规检查 → 有风险立即推送告警
   - 转写结果实时推送前端
4. 停止转写：客户端发送stop_transcription
   - 处理剩余缓冲
   - 保存完整音频文件
   - 快速分析（关键词/主题/摘要/情绪/合规）
   - 保存会议记录到数据库
   - 返回最终结果

4.6.3 技术要点

- 采样率：16kHz（浏览器端自动降采样）
- 块大小：5秒处理一次，平衡延迟与效率
- 音频保存：所有原始音频数据缓存，结束时一次性写入WAV
- 时长计算：总采样点 / 采样率，避免显示0分钟
- 音量检测：maxVolume < 0.01时提示检查设备
- 初始化顺序：Socket连接成功 → 模型加载 → 启动录音

4.7 会议检测模块（meeting_detector.py）

基于音频RMS能量的会议状态自动检测：
- 语音阈值判断：连续语音≥3秒判定为发言
- 会议开始：连续语音超过阈值
- 会议结束：连续静音≥60秒
- 最短会议：60秒，过滤短语音
- 参会人数统计：基于说话人分离结果
- 参与度分布：各说话人发言时长占比

4.8 文件解析模块（file_parser.py）

知识库文件导入支持：
- .txt：纯文本，按行解析
- .pdf：PyPDF2或pdfplumber提取文本
- .docx/.doc：python-docx提取文本
- 自动识别标题、内容、关键词等字段

七、模型体系与降级策略

7.1 设计原则

- 可用性优先：任何模型加载失败都有降级方案
- 渐进式质量：从高级模型到基础算法逐级降级
- 自动切换：异常自动捕获并切换，无需人工干预
- 离线可用：核心模型支持本地加载，不依赖网络

7.2 降级链路

7.2.1 摘要生成降级链

Qwen2.5-1.5B-Instruct → mT5_multilingual_XLSum → TextRank(LSA) → TextRank(自研)

每一级失败条件：
- Qwen → mT5：模型文件不存在 / 加载失败 / 生成失败 / 幻觉检测不通过
- mT5 → LSA：模型下载失败 / 加载失败 / 生成失败 / 幻觉检测不通过
- LSA → 自研TextRank：sumy导入失败 / 运行异常
- 自研TextRank：保底方案，纯Python无依赖

7.2.2 语义相似度降级链

bge-small-zh-v1.5 (sentence-transformers) → jieba分词 + Jaccard相似度

降级触发：
- sentence-transformers导入失败（如torchcodec问题）
- 模型文件不存在且无法下载
- 编码过程异常
- 输入文本为空

7.2.3 说话人分离降级链

pyannote.audio → 谱聚类 → GMM+层次聚类 → KMeans → 返回空结果

降级触发：
- 无HF_TOKEN / pyannote不可用
- 谱聚类失败（如样本量不足）
- 各种聚类算法依次失败

7.2.4 主题分析降级链

bge语义相似度（10s超时） → 关键词匹配

7.3 模型加载优化

- 懒加载：首次使用时才加载，加速服务启动
- 全局缓存：加载后缓存到全局变量，避免重复加载
- 离线优先：优先从本地models/目录加载，减少网络依赖
- 国内镜像：HuggingFace下载使用hf-mirror.com镜像

7.4 常见问题与解决方案

7.4.1 torchcodec导致sentence-transformers加载失败

问题：torch 2.12.0自动加载torchcodec 0.14.0（视频解码器），但找不到FFmpeg DLL，异常导致sentence-transformers整体加载失败。

解决：pip uninstall torchcodec -y（sentence-transformers不需要视频解码库）

7.4.2 HuggingFace下载慢/失败

解决：配置环境变量 HF_ENDPOINT=https://hf-mirror.com

7.4.3 模型占用内存过大

解决：
- 仅加载需要的模型（按需懒加载）
- 使用更小的模型版本（如small代替medium）
- CPU推理（device_map='cpu'）
- 量化：可进一步使用int8/int4量化减少内存

十、已知问题与优化方向

10.1 已知问题

1. 说话人分离在TTS合成音频上效果有限（音色过于接近），真实录音效果更好
2. Qwen2.5摘要CPU运行较慢（30-40秒），GPU可大幅加速
3. 实时转写使用small模型，准确率略低于medium
4. 知识库语义匹配需要人工维护关键词，初期配置成本
5. 前端为原生JS单页，交互体验有优化空间

10.2 优化方向

1. 说话人分离：集成wespeaker等中文声纹模型，提升分离准确率
2. 摘要加速：使用int4/int8量化或更小模型，或流式生成
3. 实时转写：集成Paraformer等端到端实时模型，降低延迟
4. 知识库：增加自动摘要和关键词提取，降低录入成本
5. 前端：使用Vue/React重构，提升交互体验
6. 部署：Docker容器化，支持横向扩展
7. 并发：引入任务队列（Celery），支持批量分析
8. 多语言：扩展英文/粤语等更多语言支持
